# Slice 1 PySpark Notebooks

End-to-end ETL for the `Style-channel (quantity)` PBI page on the new analytics platform.

## What these notebooks do

| # | Notebook | Builds | Depends on |
|---|---|---|---|
| 01 | `01_build_dim_date.py` | `dim_date` | `dim_date.parquet` in Volume (generated by `scripts/generate_dim_date.py`) |
| 02 | `02_seed_dim_channel.py` | `dim_channel` | `docs/data_modeling/dim_channel_seed.sql` |
| 03 | `03_build_dim_product.py` | `dim_product` | Latest `ers_product_master_YYYYMMDD.csv` in Volume + Shopify `order_line` |
| 04 | `04_build_fact_orders_line.py` ⭐ | `fact_orders_line` | All three dims above + Shopify + TW |

## Run order

```
01 ─┐
02 ─┼─► 04
03 ─┘
```

Notebooks 01, 02, 03 are independent — run in parallel. 04 hard-depends on all three.

## Idempotence

All four notebooks are idempotent: re-running them produces the same result without
duplication. Notebooks 01 and 04 use `mode("overwrite")`. Notebook 02 uses TRUNCATE+INSERT
in the seed SQL. Notebook 03 uses `mode("overwrite")` for full SCD1 rebuild.

## Smoke vs full mode

Notebook 04 supports `RUN_MODE = 'smoke'` (last 24 hours) or `'full'` (full slice 1 window).
Run smoke first on Day 3 morning, then full in the afternoon.

## Validation

Each notebook ends with post-write assertions (row counts, key uniqueness, sample inspection).
Reconciliation against legacy Panoply is performed after notebook 04 completes — see
`docs/architecture/slice_1_design.md` §9.2 for the query.

## Slice 2+ reuse

Slice 2 (`Style-channel revenue`) reuses notebooks 01–04 unchanged. Only a new YAML metric
is added in `metrics-service/metrics/`. Slice 3 (refunds) will add a 5th notebook for
`fact_refund_line`. The dim tables remain conformed.
