# 🌟 North Star: 项目最高决策原则

> 这份文档是项目的"北极星"。
> **任何决策冲突时,以这份为最高优先级。**
> Claude 在每次给建议前,必须先用这些原则过滤一遍。

---

## 🎯 第一原则:最大化回国找 DE 工作的简历价值

这个项目**首要目的是简历**,其次才是 leader 的需求。

### 核心含义

1. **任何技术选型,先问"对回国找 DE 工作加分多少",再问"是否最快完成"**
2. **如果某个决策只满足 leader 但不加分简历,要找平衡方案**
3. **如果某个决策加分简历但 leader 不在意,主动推动**(只要不影响交付)
4. **简历素材的密度 > 项目完成速度**

### 实操原则

| 场景 | 决策方向 |
|---|---|
| 选 A 简单但简历素材弱,选 B 难但简历素材强 | **选 B**,除非 B 严重影响交付 |
| Leader 想要快速实现,但快速方案没有"DE 关键词" | 找折中方案,既能交付又能写关键词 |
| 加一个功能多花 2 天,但能多 1 个简历 bullet | **加** |
| 简化一个设计能省 1 周,但简历少一个亮点 | **不简化**,除非时间真的来不及 |

### 求职目标

- **目标档位**:中级 / 中级偏高 Data Engineer
- **薪资带宽**:35-50K(冲刺 50-60K)
- **目标公司**:国内中大厂、外企在中国分部、成熟创业公司

## 🎯 第二原则:技术深度 > 技术广度

**一个能讲 45 分钟的项目 > 三个浅项目**

### 含义

1. 不做"多个浅项目"——专注把这一个项目做透
2. 每个 Phase 都要有"能讲 5 分钟的硬核技术点"
3. 关键词清单(必须命中):
   - 数仓分层(ODS/DWD/DWS)
   - Kimball 维度建模(fact + dim)
   - PySpark ETL
   - 指标平台 / 数据服务化
   - Data Contract / Schema Governance
   - Versioned Metrics / Metric Governance
   - Data Quality / Data Lineage
   - Orchestration(调度编排)
   - Caching(性能优化)
   - Streaming + Watermark + Exactly-once
   - 流批一体(Unified Streaming + Batch)
   - 成本优化(FinOps)

### 检查清单

每完成一个 Phase,问自己:
- [ ] 我新增了哪几个简历关键词?
- [ ] 这些关键词能讲清楚 trade-off 吗?
- [ ] 面试官追问 3 层深度,我能扛住吗?

如果有"扛不住"的,补强后再进入下一个 Phase。

## 🎯 第三原则:工程化思维 > 单纯技术堆砌

**面试官追问的是"为什么这么做",不是"做了什么"**

### 含义

1. **先写设计文档,再写代码**(Design before Code)
2. **每个决策有 trade-off 分析**(Decision Log)
3. **做事留痕**(每个 Phase 有总结文档)
4. **量化结果**(节省 $X、降低 N%、服务 M 用户)

### 必做实践

- ✅ Data Contract(数据接入前)
- ✅ Schema 设计文档(建表前)
- ✅ Architecture Design Doc(每个 Phase)
- ✅ Decision Log(`PROJECT_CONTEXT.md`)
- ✅ Phase 总结文档(每个 Phase 完成后)
- ✅ 量化所有数字(成本、性能、用户数)

## 🎯 第四原则:满足 Leader 是底线,简历是上限

**Leader 的需求是"必须满足",简历的需求是"主动追求"**

### 实操

| 维度 | Leader 需要 | 简历需要 | 如何平衡 |
|---|---|---|---|
| 速度 | Phase 1 MVP 越快越好 | 不能太糙 | Phase 1 用 mock 数据快速交付,后面持续升级 |
| 替代 PBI | 看到能用的网页 | 端到端数据平台 | 用网页做"展示前端",底下是完整数据平台 |
| 成本节省 | 省订阅费 | 量化成本数字 | 主动算 ROI 报告 |
| 团队能用 | N 个用户用上 | 有用户使用统计 | 加埋点统计使用情况 |

### 反例(不要做)

❌ "Leader 没要求,我也不做"——错过简历素材
❌ "Leader 要求很急,简化设计"——除非真的来不及
❌ "Leader 不懂技术,我堆砌技术栈"——技术服务业务,不能本末倒置

---

## 🎯 第五原则:基于反馈迭代,不要焦虑

**项目做完投简历测试,根据反馈决定下一步**

### 含义

1. **不要做"防御性堆砌"**(怕简历不够,什么都加)
2. **Phase 6 完成后投一波简历**,看反馈
3. 根据反馈决定:
   - 反馈好 → 继续刷算法/八股
   - 反馈"缺实时" → 已规划 Phase 4.5
   - 反馈"缺大数据量经验" → 这个项目没法补,转向工作经验
   - 完全过不了筛选 → 简历包装问题,改简历不改项目

### 不焦虑

- ✅ 这一个项目做透,够中级 DE
- ✅ 加 Phase 4.5 实时模块,够中级偏高
- ❌ 不要因为焦虑做第二个项目(浪费时间)
- ❌ 不要因为焦虑做"双壳配置"(国内不吃这套)

---

## 🚨 红线(绝对不能做)

1. **不要为了交付速度牺牲简历核心素材**
   - 例:不要因为快用 Next.js 直连 Databricks(这会丢掉 FastAPI 服务层这个简历核心)
2. **不要做"国内 + 海外"双套配置**
   - 浪费时间,简历加分有限
3. **不要中途换技术栈**
   - 已锁定的 Databricks / FastAPI / Next.js / YAML DSL 不要再翻盘
4. **不要"为了写关键词"做不合理的设计**
   - 例:为了写"实时"硬接 Kafka(简化路径用 Auto Loader 就够)
5. **不要忽视量化**
   - 每个 Phase 完成后必须有"省了多少钱、提升多少 X"的数字
